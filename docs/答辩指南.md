## 三、答辩 QA 预判

### Q1：为什么用 Shell 而不是 Python？

**答**：Shell 是 Linux 的原生语言，直接与内核接口交互（`/proc`、系统命令），没有中间层。课程设计的目标是深入理解操作系统内核机制，Shell 让你直面系统调用和内核接口，而不是通过高级语言的封装。另外 Bash 脚本在运维场景中部署零依赖，是 DevOps 工程师的必备技能。

### Q2：/proc 文件系统是什么？你们怎么用的？

**答**：`/proc` 是 Linux 内核提供的虚拟文件系统，不占用磁盘空间。它把内核数据结构以文件的形式暴露给用户空间。我们通过读取 `/proc/stat` 获取 CPU 时间片分配，通过 `/proc/meminfo` 获取内存页框使用情况，这些都是内核调度器和内存管理子系统的直接反映。

### Q3：CPU 使用率怎么计算的？

**答**：使用**两次采样差值法**。`/proc/stat` 第一行记录了系统启动以来的累计 CPU jiffies（时间片）。我们取两次采样（间隔 0.5 秒），计算 delta_total 和 delta_idle 的差值，得到这段时间内的 CPU 利用率。各核心同理，读取 `cpu0`, `cpu1` 等行。

### Q4：你们的暴力破解检测是什么原理？

**答**：解析 `/var/log/auth.log` 中 "Failed password" 行，提取时间戳和来源 IP。设定 300 秒时间窗口，同一 IP 失败超过 5 次就标记为暴力破解。这是一个简化的入侵检测系统（IDS）思路，实际生产环境会结合 fail2ban 等工具。

### Q5：SUID/SGID 是什么？为什么要检查？

**答**：SUID (Set User ID) 和 SGID (Set Group ID) 是文件的特殊权限位。SUID 文件执行时以文件所有者（通常是 root）身份运行，是 Linux 中提权的常见途径。`passwd` 命令就是典型的 SUID 程序。我们检查非系统目录中的异常 SUID 文件，防止后门提权。

### Q6：管道（pipe）在项目中怎么用的？

**答**：模块四的实时日志追踪用到 `tail -f | grep --color`，这是一个典型的管道应用——前一个命令的 stdout 直接作为后一个命令的 stdin，内核通过管道缓冲区实现进程间通信（IPC），数据不需要落盘。模块三中 `find ... | xargs tar` 也是管道通信。

### Q7：trap 命令做什么的？

**答**：`trap` 是 Shell 的信号处理机制。我们在实时日志追踪（Ctrl+C 停止）和守护进程中使用 `trap 'cleanup' SIGINT SIGTERM` 来捕获中断/终止信号，做优雅退出——清理临时文件、关闭日志文件句柄，避免资源泄露。

### Q8：守护进程你是怎么实现的？

**答**：使用 `nohup` 让进程忽略 SIGHUP 信号 + `&` 后台运行 + PID 文件记录进程号。主循环是 `while true; do 巡检; sleep $interval; done`。通过 PID 文件实现生命周期管理（启动前检查是否已运行，停止时 kill PID）。

### Q9：为什么 dialog 菜单和纯文本菜单两套？

**答**：兼容性考量。dialog 需要额外安装，且在某些最小化安装的服务器上不可用。我们做降级策略：有 dialog/whiptail 就用图形 TUI，没有就降级到纯文本菜单，保证在裸机 Ubuntu Server 上也能完整运行。

### Q10：如果要扩展，你们会加什么？

**答**：三个方向：(1) 网络监控模块，`/proc/net/dev` 流量统计；(2) 告警通知，集成 Telegram Bot 或邮件发送；(3) Web 面板，用 Python Flask 做前端展示替代 HTML 静态报告。

## 四、给 PPT 制作人的建议

1. **截图要清晰**：在 WSL 终端运行后截图，使用 Windows Terminal 的黑色背景，字体调大到 14px+
2. **代码不用全放**：选关键函数各放 1 张即可（如 CPU 差值算法、暴力破解检测逻辑）
3. **架构图放 C 位**：从设计文档中取架构图，这是答辩老师最看重的
4. **配色建议**：深色底 + 蓝/绿色调，代码用等宽字体
5. **动画慎用**：技术答辩用静态图更清晰，翻页过渡简单即可
